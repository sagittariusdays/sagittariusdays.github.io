梅雨随笔

发表于 2020-07-06 分类于 essay

梅雨狂妄

7月6日，今年的梅雨季节似乎比往年要长一些。

梅雨锋恣意地在日本列岛南北蹂躏，九州地区的降雨导致了河川水位上涨，河流决堤——然而也并没有什么堤好决的，似乎是他们一开始就反对建造防洪治水大坝的样子。

这又不能说自讨苦吃自作自受，毕竟受灾了也是受灾了。但是在可以防患于未然的情况下却不去做，这就有点让人无话可说了。

人吉所在的小盆地被淹，学校里地面上贴出“SOS”求救信号，城镇水位上涨到了8m到9m，一副98年的样子。

也正是前几日，南方，广西湖南也遭受了洪灾。这一幕幕场景就在今天，又仿佛是在22年前。被洪水冲垮的房屋，冲散的群众，这么多年来，似乎从未变过。

哪里的受灾都不值得嘲笑，一如经典的那句话：在大自然面前，人类才是渺小的。

梅雨锋

日语中叫作梅雨前線。其成因，简而言之就是在由春转夏的时期，南方海洋上的高压气团北上，与本来温带的高压冷气团相僵持，长时间在同一地点形成静止锋。南方的暖气团被迫抬升，从而导致对流活动，出现降水。

一般而言这个静止锋，在大陆则是西伯利亚高亚北去之后形成的热带大陆性移动气团，日语称长江气团，与来自印度洋的副热带高压相汇合产生降水。6月时位置大概在华南岭南，随后逐渐北移至江淮一带。日本方面则是由被青藏高原分离的两股高空湍流在千岛——堪察加一带汇聚而成的鄂霍次克气团与太平洋高气压所汇合产生伴随日本列岛的一条梅雨锋。

这条锋面从6月开始上陆日本，在南北两方反复的拉锯之下逐渐北上。七月上中旬，由于高空湍流完全绕至青藏高原北边，由此导致汇合产生的鄂霍次克气团减弱乃至消失，日本的梅雨季节就此结束。

然后秋雨的时节再反过来走一遭。

极端的灾害

近年极端的灾害正在增加。以前并没有如此多的报道，然而前几年广岛土砂灾害，随之而来的又是去年的大量降雨，台风势力也年年增强。以往五十年一遇的灾害正渐渐变成日常。

的确，全球气候变暖可能比我们所认知的更严重。南北两极冰川融化，水面上升，夏季海洋温度升高，从而导致暖湿气团带有更多水汽，也使产生台风的动力更加充足。这每一个小课题都值得总结一篇小作文。

同时如何在这样的情况下生存下去，如何适应、对应，也是十分值得思考的。

这样写着，我关上电脑，准备睡去。

明天的梅雨，似乎比这一个月的都更严重。

火神山雷神山建设之思考

发表于 2020-07-05 更新于 2020-08-16 分类于 excerpt

本文主要参考了维基百科——火神山医院页面。如与事实有出入，以最新消息为准。

10天完工的大型医院：概述

2020年初，武汉发现了一种新型的冠状病毒，截止到半年后的7月5日今天，全世界已经有超过1100万人感染，53万人因病去世，并且在疫情比较严重的国家比如美国，巴西，俄罗斯，疫情依然在大幅扩散。1

面对这次疫情，武汉在1月23日，农历腊月廿九，宣布兴建武汉火神山新型冠状病毒感染肺炎专科医院，又称火神山医院，参考了17年前北京小汤山非典医院的模式，用于专门收治2019年新型冠状病毒感染的病患。

医院于1月25日大年初一开始兴建，施工工程由中建三局牵头，2月2日完工，交由中国人民解放军联勤保障部队（军队援鄂抗疫医疗队），2月3日进行接收推演，2月4日接收首批感染病患。2

工程实际施工9日，建筑总面积3.39万平米3，设有床位1000张，设计依据《传染病医院建设标准》设立，并保有相关级别的传染病防控措施，医疗废水并不会排入邻近的水体，而是收集起来进行消毒处理之后统一排放，院内设有由华为提供的5G网络4，满足了各级的需要。医院于2020年4月15日完成使命关闭。

工程上的思考

设计

医院参考了2003年非典时期建于北京的小汤山医院模式，承担火神山医院设计任务的是中信建筑设计研究总院5。

时间上，1月23日上午10时武汉市宣布封城，当日下午决定建设医院，同时中信总院承接了设计任务。场地平整设计图于5小时之内完成，随即医院连夜开工。基本方案设计图于24小时内通过武汉市政府认可，1月26日凌晨全部施工图按时交付6，整体上工程时间为60小时左右。

设计团队上，除了建筑，结构，给排水，暖通空调，电气等的专家设计师以外，包括后勤保障人员，共计60余人组成的医疗建筑上经验丰富的设计团队参与了设计。

设计采用了多个“H”型模块排列而成，每50个床位为一个护理单元，需要配备的护士，医生，医疗器械，供氧系统等均由护理单元来进行计算。每4个护理单元形成一个组团，组团连接的“H”字中心部为公共区域，供医护人员工作。7
已公开的图纸大致如下：

设计上的使用寿命为5年，到期之后可以通过加固和重新检验来延长使用寿命，但是作为临时的传染病医院，一般不这样做。事实上，火神山雷神山两家医院均于4月中停止使用，实际使用寿命并没有超过三个月。

如此模块式的医院，在未来可以通过3D打印来实现，不过现阶段来说这个技术受限于材料和打印机的限制，成本过于高昂。

施工大致

施工上，除了中建三局以外，第一时间共有12家单位参与，从各地集结了4万多名管理施工人员，共投入了3500多台机械设备。8

日本网站上总结的工程表：9

事实上，整地工程自1月23日，即前述的场地平整设计图完成之后即施工开始。这样在继续设计的同时，工程已经开工，争取了宝贵的时间。

此外，由于这种施工的特殊性，需要大量的城市维生管线接入，同时对施工材料，器械的入场便利性有特殊的要求，再加上全部智能医院的所需，医院选址于城市近郊。

整体医院属于预计短期使用的预制件组装房，地基施工只基本只包括混凝土地面浇铸，不需要打地基桩，因此在施工上可以对地基沉降等的要求稍微低一些。

在施工的高效化上，每相邻的两个病房之间保9.6m的间距，在这个空间内可以令施工重机进入，多个机械同时作业提高了工作效率。

当然，还有众所周知的 “视频直播施工”，在全世界的注视下，24小时轮换不停地进行施工，从而保障医院按期完工。

换气设备10

由于使用性质为传染病医院，所以选址位于武汉主导风向东北风的下风向，并且在医院的下风向为水体，基本保障了医院的污染对于城市的影响最低。

设计上，考虑到武汉冬季的气温较低湿度较大，医院设计在考虑防潮的同时，采用了分体空调，在进风段设置了加热装置，通风管道也进行了保温施工。

在设计分区上，清洁区，半污染区与污染区分开，保证了清洁的医护通道，一般的医护通道与病人的动线区分开，中间设有缓冲区，不直接交叉。

此外在通风系统上，清洁区域，半污染区与污染区分别设置独立的通风系统，保障办公室，值班室等清洁区为正压，护士站，处置诊疗室等半污染区为微负压，病房及病员通道等污染区为负压，从而保证了气流的正向流动，没有交叉污染。相邻的区域的压差不小于5Pa。

除了保证压力之外，医院还对空气的过滤有很高的要求，具体此处不再详述。

通信设备11

1月23日施工开始后，三大运营商接到通知，随即开展覆盖两家医院的4G、5G网络工程建设。湖北移动、湖北联通武汉分公司与华为公司和设计院等单位一起，立即开始规划，设计和物资的协调。

24日除夕上午现场勘查完成，建设方案确定，下午完成设备物资等的调配，当晚即开始施工。

25日大年初一上午，完成光纤铺设、基站天线架设和主设备安装等工作，下午完成了基站的开通和调测，晚上则完成了业务测试和网络优化工作。

整个5G网络从规划到落地共用了仅3天的时间。

此外，华为向火神山医院捐赠了80%的网络通信设备，包括无线AP、接入网络、汇聚网络、核心网络、视频监控和远程会议系统等共几千台设备，提供了高标准的CloudCampus智简园区的方案，在保证整个医院无死角的Wi-Fi覆盖，同时也满足了视频监控等大量需要的带宽。

关于IoT，所有病房里安装了大量的物联网设备，对于药房，进行其温湿度监控；对于隔离病房里的病患也进行了监控，并配有移动护理的腕带，保障随时了解其情况；对于传染垃圾也有IoT设备进行追溯。从入到出不仅为医院的管理提供了便利，保证了服务质量和效率，从病患监控的设备上亦可以对个人状况进行实时的数据分析，以便做出更佳的决策。

其他

国家电网进行施工，1月31日夜，医院正式通电。

医院配有近4万平米的绿化建设，主要为景观绿化，设计于1月29日完成。当时有质疑的声音说绿化可能导致病毒下渗至土壤，现在已经找不到原文。

由于医院的特殊性，需要大量的医疗用氧，以及输氧管线。之前有印象看到过总结文章，但是现在没有找到。 医疗用氧由武钢提供，将原来工业用氧的生产线抽调制作医疗用氧，管线调配暂不清楚。

统率全局的PMr9

在保障这个大工程时，能够调配所有的资源的PMr(Project Manager)起到了核心的作用。

为了保证以下工程顺利进行，总指挥调配所有能够支配的资源，并且保证各级，各层的沟通，工程的进行。能够胜任这个位置的，需要有足够的能力与魄力，也要承担足够的责任与压力。

此处工作的描述实际参照日文网站，实际上在公开搜索渠道能够查到的工程总指挥是中建三局的党委副书记、总经理陈卫国。 12

感想

稀里糊涂零零散散地总结了一气，现在依然是心潮澎湃，充满感动。自己的知识储备不足，也不足以描述整个过程中我感受到的冲击。虽然这次是在准备资料总结建设业在Covid-19影响下、影响后能做些什么，但是写完这篇我心里实际上知道，这个成功几乎是没法复制的。

整个建设业在疫情影响之后能做的事情，或许与疫情之前能做到的事情差不太多。一直以来的思考是，建设业需要在一切既往的工程上面加上BIM/智能Robot施工等用来削减人工的应对措施，然而在紧急事态上，也是依然有需要调动足够大量的人工的需要的。

对于自己来说，能够从这些资料中学到什么，能够把这些东西在之后的工作生活中活用多少，才是真正有意义的总结。

于2020年7月5日
傍晚。

关于口罩

昨天终于在Muji买了些布口罩。
我之前一直是认为必须要VFE的口罩的。
布口罩顶多也就是防一个飞沫粉尘细菌，病毒什么的效果也不大。

但是自从家里囤货的VFE口罩见底了，我开始用上普通口罩之后，思想就略微产生了变化。

总归戴口罩，可以隔离大量的飞沫，危险性比起什么防护都没有的时候，已经有一些了。更何况大家都戴口罩，空间里的飞沫量会相应少很多，如果有，也就是飞沫附着在口罩表面，然后再被手摸上去，被传染吧。

这样想来，如果一直呆在有病毒的空间里，本身眼睛什么的，也是有可能传染的。

这种普通的口罩，防止一般的传染就够了。如果想防止Covid-19这种似乎无孔不入的病毒，只要大家不是100%地做到各种级别全面的防护，比如橡胶手套护目镜防护服，不与外界做任何接触，也不是100%安全的。

现在倒是能理解之前号召的，普通群众只需要一般的医用口罩即可，N95级别的口罩尽量留给医护人员，这样的需求了。

首页与博客分离的方法

发表于 2020-06-22 分类于 Hexo

虽然还没有试验，但是应该可行的一个首页与博客分离的方法。

原理

现在的博客是部署在yourname.guthub.io上面，直接访问yourname.github.io则会指引到博客的首页。
现在我并不想直接在入口处显示文章，而是想做一个封面式的入口，在首页上可以选择是哪一个博客，甚至可以将技术博客与散文博客分开，这样子用目前的Hexo的Next主题并不是很好操作。

之前我制作过几个保存其他文件的repository，它也有一个页面，并且是在yourname.github.io/repositoryname的子页面下，它的上一层页面正是我现在的首页，即放上了文章的博客首页。
那么可不可以在名字下面建立blog，technical blog等几个repository，然后在yourname.github.io页面上只做一个入口index.html，可以选择进入blog或者technical blog的页面呢。

目前我觉得是可行的，那么这一周的目标就是找合适的方法制作出一个index.html,然后再将现在的内容分别部署至多个repository去。
